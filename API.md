# CollabV AI ‚Äî Backend API Reference

CollabV AI is a pure REST API service ‚Äî there is no bundled UI. It's built to
be consumed by a separate frontend (the production CollabV website lives in
its own repository, not this one) or by any other authorized client.

This document is a curated, engine-oriented index. The full, always-accurate,
machine-readable spec is auto-generated by FastAPI and available from the
running service itself:

- **Interactive docs (Swagger UI):** `GET /docs`
- **Raw OpenAPI 3 spec:** `GET /openapi.json`
- **ReDoc:** `GET /redoc`

If this file and the live `/openapi.json` ever disagree, the live spec is
correct ‚Äî it's generated directly from the route definitions, this file is
maintained by hand.

## Live-data-only

Every entity type below (professors, patents, companies, problem statements,
students, employees, institutes, job postings, research opportunities) starts
empty and is populated exclusively through this API by real registered users.
There is no seed/sample/demo data loaded at startup. See `professors_live.json`
(an intentionally empty `[]`) and `collabv/patent_problem_db.py`'s removed
auto-seed call for how this is enforced at the code level.

## Authentication & Authorization

Two accepted credential types, checked by `collabv/security.py::require_auth_or_api_key`
and `collabv/auth.py`:

- **JWT bearer** ‚Äî `Authorization: Bearer <token>`, obtained via `POST /auth/login`
  after registering with `POST /auth/register`.
- **API key** ‚Äî `X-API-Key: <key>`, generated via `POST /auth/api-key/generate`
  (requires a valid session first).

Roles (`collabv/auth.py::VALID_ROLES`): `admin`, `company_user`, `professor_user`,
`buyer_user`, `student_user`, `employee_user`, `institute_user`. Role-gated routes
use `require_role(*roles)`; most matching/dashboard routes trust the `{student_id}`/
`{professor_id}`/etc. path parameter directly rather than deriving identity from
the token ‚Äî this is the existing platform convention, not an oversight, and applies
uniformly across every dashboard.

**`AUTH_REQUIRED=false` (dev default) disables enforcement entirely** ‚Äî every
endpoint is reachable with no credentials, which is why local development never
needs a token. **Set `AUTH_REQUIRED=true` in production** (see `.env.production.example`).

## CORS

Configured in `collabv/security.py::install_security`. In dev (no `ALLOWED_ORIGINS`
set), any `http://localhost:<port>` or `http://127.0.0.1:<port>` origin is allowed
automatically ‚Äî this is why the local Next.js dev server on any port just works.
**In production, set `ALLOWED_ORIGINS` to a comma-separated list of the exact
origin(s) the separate production website is served from** ‚Äî nothing is allowed
by default once that variable is set.

## Configuration reference (environment variables)

All server behavior is environment-variable-driven ‚Äî no hardcoded hostnames,
ports, keys, or feature flags. See `.env.example` (dev) and
`.env.production.example` (prod) for copy-paste templates.

| Variable | Purpose | Notes |
|---|---|---|
| `PORT` | Server port | default `8000` |
| `DEBUG` | FastAPI debug mode | **never `true` in production** ‚Äî leaks stack traces |
| `DB_FILE` | SQLite database file path | default `collabv_data.db` |
| `PROFESSORS_FILE` | Base professor directory JSON | must stay `professors_live.json` (empty) ‚Äî live-data-only |
| `ENABLE_EMBEDDINGS` | Toggle dense/semantic matching | needs `sentence-transformers` (~80MB model) |
| `ENABLE_LLM_EXPLAIN` | Toggle Claude-generated explanations | falls back to rule-based text if `false` or no API key |
| `ANTHROPIC_API_KEY` | Claude API key | optional ‚Äî every AI-text feature has a rule-based fallback |
| `JWT_SECRET` | JWT signing secret | **required** if `AUTH_REQUIRED=true`; generate with `secrets.token_urlsafe(64)` |
| `AUTH_REQUIRED` | Enforce auth on protected routes | `false` in dev, `true` in production |
| `ALLOWED_ORIGINS` | CORS allow-list | comma-separated; required in production |
| `RATE_LIMIT_MATCH_PER_MIN` / `RATE_LIMIT_MATCH_BURST` | Per-IP rate limit on matching endpoints | in-memory limiter |
| `RATE_LIMIT_CONTRACT_PER_MIN` / `RATE_LIMIT_CONTRACT_BURST` | Per-IP rate limit on contract endpoints | in-memory limiter |
| `DATABASE_URL` / `DATABASE_HOST`+`DATABASE_USER`+`DATABASE_PASSWORD`+`DATABASE_NAME`+`DATABASE_PORT` | Postgres connection (future) | `collabv/db_postgres.py` + `alembic/` exist for this migration but aren't wired into the live request path yet ‚Äî the active DB layer is SQLite via `DB_FILE` regardless of these |
| `SENTRY_DSN`, `REDIS_URL`, `LOG_LEVEL`, `ENVIRONMENT`, `COLLABV_IITM_PATENT_API` | Reserved | not read by any code yet; placeholders for future observability/caching work |

## Matching engines ‚Üí REST endpoints

Every AI model / matching engine in `collabv/` is reachable purely through
REST ‚Äî there is no engine that's only invokable as a Python import.

| Engine | Module | What it does | Endpoints |
|---|---|---|---|
| 1 | `matching_engine.py` | Professor‚ÜîCompany, 3-tier keyword+embedding+soft-filter scoring (the original engine) | `POST /company/request`, `POST /match/run`, `GET /match/results/{id}`, `GET /match/results/{id}/explain`, `GET /professors`, `GET /professor/{id}` |
| 2 | `matching_engine_2.py` | Professor ‚Üí Company (reverse direction) | `POST /professor-match/run`, `GET /professor-match/companies` |
| 4 | `matching_engine_4.py` | Company (profile + problem statement) ‚Üí Professors & Patents | `POST /matching-engine-4/run`, `GET /matching-engine-4/problem/{id}/matches`, plus the `CompanyDashboard` group |
| 5 | `matching_engine_5.py` | Unified engine (merged former 3/5/6): patent‚Üîaudience matching across company/professor/student/employee/institute | `POST /match` (unified), `POST /marketplace/discover/{buyer_type}/{buyer_id}/grouped`, `POST /match/interactions` |
| 7 | `matching_engine_7.py` | Student/Employee ‚Üí Patents & Professors | `POST /student-dashboard/{id}/recommendations`, `POST /employee-dashboard/{id}/recommendations` |
| 8 | `matching_engine_8.py` | Research Opportunities: Student/Employee ‚Üî Professor-posted positions (PhD/Master's/internships/fellowships/etc.) | see **MatchingEngine8** table below (21 endpoints) |
| 9 | `matching_engine_9.py` | Job Postings: Student/Employee ‚Üî Company job postings | see **MatchingEngine9** table below (9 endpoints) |

(Former Engines 3 and 6 were merged into Engine 5 ‚Äî see that module's docstring
‚Äî so there's no standalone Engine 3/6 file or endpoint group.)

Every non-"matching engine" AI feature is likewise REST-reachable, either
directly or through the engine that calls it: `explainer.py` (match
explanations), `deal_scorer.py` (`POST /deal/score`), `contract_nlp.py`
(`Contracts` group), `skill_gap_analyzer.py` (`.../skill-gap/{id}`),
`startup_insights.py` (`.../startup-insights`), `collab_readiness.py`
(`.../readiness`, `/readiness/departments`), `knowledge_graph.py` (`/kg/*`),
`retrainer.py` (`Admin` group), `resume_parser.py`/`resume_storage.py`
(resume upload/download).

## Auto-refresh semantics (Matching Engines 8 & 9)

Job and research-opportunity match scores are **not** recomputed on a
schedule or in a background worker. They're recomputed lazily, synchronously,
on read: every `GET .../job-matches` or `GET .../opportunity-matches` call
compares the cached score's `profile_version`/`job_version` (or
`opportunity_version`) against the live `updated_at` timestamps on the
student/employee profile and the posting, and recomputes+upserts anything
stale before returning. Editing a resume or a posting therefore reflects in
the very next read ‚Äî no polling or webhook needed on the consuming
frontend's side.

## Error format

All errors use a stable `{"error": "<CODE>", "message": "<human text>"}` shape
via `collabv/errors.py::api_error` (see the `ErrorCode` enum for the full list
of codes and their HTTP status).
